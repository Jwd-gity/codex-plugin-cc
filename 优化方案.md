## P0: 子 Agent 输出验证规则

### 问题

子 Agent 返回"代码已写入"/"测试通过"后，主 Agent 直接接受，无独立验证。

**AutoMAS 证据**:
- Lesson 007: Executor 偷懒 + Checker 幻觉 = 系统性虚假高分
- AUDIT T-004: "lessons not consulted before experiments" — 系统重复已知错误
- history.json: evo_001 有 4 次 score=0，无错误记录，无事后分析
- evo_round1.log: 运行到 task 7 崩溃，但 history.json 只记录 score=0，无崩溃原因

**ECC 当前状态**: PostToolUse hooks 为空数组。Agent 工具返回后无任何验证逻辑。

### 方案

修改 `~/.claude/rules/zh/agents.md`，新增:

```markdown
## 输出验证（强制）

子 Agent 声称完成操作后，主 Agent 必须独立验证：

| 声称 | 验证方式 | 判断依据 |
|------|----------|----------|
| "文件已写入" | Read 文件确认存在且内容非空 | 文件字节数 > 0 |
| "测试通过" | Bash 运行测试命令 | exit code == 0 |
| "构建成功" | Bash 运行构建命令 | exit code == 0 |
| "代码已修复" | Read 文件确认修改已应用 | diff 内容匹配 |
| "安装完成" | Bash 检查 package/命令 | which/ls 确认 |

**禁止**: 直接接受子 Agent 的完成声明而不验证。
**例外**: 纯研究/分析任务（无文件系统操作）可跳过。

**原因**: AutoMAS 实验证明，不验证会导致 Agent 偷懒+幻觉串通，
系统性产出虚假高分但实际无有效输出。
```

### 影响

- 文件: `~/.claude/rules/zh/agents.md`
- 工作量: 5 min
- 验证: 派遣子 Agent 写文件，观察主 Agent 是否 Read 验证

## P1-A: 代码审查单轮限制

### 问题

development-workflow.md 第 3 步无迭代上限。code-reviewer → fix → re-review → re-fix 循环。

**AutoMAS 证据**:
- Lesson 001: 代码任务自反射让 Gen 从 68.6 暴跌至 <20
- TEAM_CONTEXT.md: "Code Agent 禁止自反射（已验证会导致灾难）"
- v37.0 扩展自评审 → 分数从 76.22 降至 69.07

**类比**: 代码审查-修复-再审查 = 代码任务的变相自反思。

### 方案

修改 `~/.claude/rules/zh/development-workflow.md` 第 3 步:

```markdown
3. **代码审查（单轮）**
   - 编写代码后使用 **code-reviewer** 代理（仅一次）
   - 解决 CRITICAL 和 HIGH 级别问题后完成
   - 不对修复后的代码再次审查（除非用户明确要求 "再审查一次"）
   - 理由: AutoMAS 证明代码任务的反复审查-修复循环会导致质量退化
```

修改 `~/.claude/rules/zh/code-review.md` 末尾新增:

```markdown
## 审查迭代限制

**代码审查限单轮执行。** 发现问题 → 修复 → 完成。

原因: 反复审查-修复循环等同于代码任务的自反思（AutoMAS Lesson 001），
会导致质量退化而非提升。v37.0 扩展自评审实验: 76.22 → 69.07。
例外: 用户明确要求时可执行第二轮。
```

### 影响

- 文件: `~/.claude/rules/zh/development-workflow.md`, `~/.claude/rules/zh/code-review.md`
- 工作量: 5 min

## P1-B: 规则主动查询机制

### 问题

ECC 有 16 个规则文件在 `rules/zh/` 下，但 Agent 不会主动查询它们。规则写了等于没写。

**AutoMAS 证据**:
- AUDIT T-004: "knowledge/lessons/Lessons_Learned.md exists but is not referenced in evolution decisions"
- AUDIT T-004: "No pre-experiment checklist that queries past lessons"
- 结果: v32 重复了 v31 已验证的 6000 tokens 边际递减教训

**ECC 当前状态**: `rules/zh/` 有 agents.md、code-review.md、testing.md 等 16 个文件，全部通过 settings 自动注入上下文。但:
- 新会话开始时不会主动检查相关规则
- 编码前不会查询 testing.md 的覆盖率要求
- 提交前不会查询 security.md 的检查清单

### 方案

在 `~/.claude/rules/zh/development-workflow.md` 第 0 步前新增:

```markdown
## 规则查询检查点（强制）

在以下时刻，必须检查相关规则文件：

| 时刻 | 必查规则 | 检查内容 |
|------|----------|----------|
| 编码前 | coding-style.md | 命名、不可变性、错误处理要求 |
| 测试前 | testing.md | 覆盖率要求、TDD 流程 |
| 审查前 | code-review.md | 检查清单、严重级别定义 |
| 提交前 | security.md | 密钥检查、输入验证、OWASP 清单 |
| PR 前 | git-workflow.md | 提交格式、PR 模板 |

不查规则就行动 = 重复已知错误。
```

### 影响

- 文件: `~/.claude/rules/zh/development-workflow.md`
- 工作量: 5 min

## P1-C: 静默失败检测

### 问题

子 Agent 执行失败时，主 Agent 可能不感知或不处理。

**AutoMAS 证据**:
- AUDIT EH-006: "JSON parse failure silently defaults to score 50" — 完全损坏的响应和 50 分正常响应无法区分
- history.json: evo_001 有 4 次 score=0，无任何错误记录
- evo_round1.log: task 7 崩溃 ("Results file not found")，但后续仍记录 score=0 而非 ERROR
- harness_evolution.py: `record_result()` 中 `if result and "summary" in result: score = ...` else `score = 0` — 错误和零分混为一谈

### 方案

在 `~/.claude/rules/zh/agents.md` 的"输出验证"章节后追加:

```markdown
## 静默失败检测（强制）

子 Agent 返回结果时，检查以下静默失败信号：

| 信号 | 含义 | 处理 |
|------|------|------|
| 返回内容为空或极短 (<50字) | Agent 偷懒或崩溃 | 重新执行 |
| 声称成功但无具体输出 | 幻觉完成 | 独立验证 |
| 返回 "无法"/"抱歉"/"不知道" | 任务失败 | 报告错误，不假装成功 |
| 错误码 0 但内容异常 | 静默失败 | 检查输出合理性 |

**禁止**: 将失败结果当作成功处理。
**禁止**: 将空输出当作"已完成"。
**原因**: AutoMAS 的 `parse_evaluator_response()` 在 JSON 解析失败时
默认返回 score=50，导致损坏的评估结果和正常结果无法区分。
```

### 影响

- 文件: `~/.claude/rules/zh/agents.md`
- 工作量: 5 min

## P2-A: Stop Hook 最终验证

### 问题

会话结束时无检查。可能留下: 未提交更改、临时文件、声称完成但未验证的任务。

**AutoMAS 证据**:
- MEMORY_SYSTEM.md: "每次 Session 结束时更新 memory/today.md + 检查未完成任务"
- MANAGEMENT.md: 归档流程 7 步，从保存结果到清理临时文件

### 方案

新建 `~/.claude/scripts/session-stop-check.sh`:

```bash
#!/bin/bash
echo "=== Session Stop Check ==="
CHANGES=$(git status --porcelain 2>/dev/null | wc -l)
[ "$CHANGES" -gt 0 ] && echo "[WARN] $CHANGES uncommitted changes"
BIG_TEMPS=$(find /tmp -maxdepth 1 -size +10M -user $(whoami) 2>/dev/null | wc -l)
[ "$BIG_TEMPS" -gt 0 ] && echo "[WARN] $BIG_TEMPS large temp files in /tmp"
echo "=== Done ==="
```

`settings.json` 新增 Stop hook 指向此脚本。

### 影响

- 新建: `~/.claude/scripts/session-stop-check.sh`
- 修改: `settings.json` hooks.Stop
- 工作量: 5 min

## P2-B: 模拟数据标记规则

### 问题

**AutoMAS 证据**:
- Lesson 004: `evolution_loop_MOCK.py` 返回 `random.uniform(-5, 10)` 假分数
- AUDIT E-003: "evolution_loop.py IS A SIMULATION — Not Connected"

### 方案

在 `~/.claude/rules/zh/coding-style.md` 新增:

```markdown
## 模拟数据标记（强制）

- 内容: 开头包含 `[MOCK DATA]` 或 `"simulated": true`
- 禁止: 真实数据格式的未标记模拟输出
```

### 影响

- 文件: `~/.claude/rules/zh/coding-style.md`
- 工作量: 2 min

## P2-D: 规则-行为一致性审计

### 问题

**AutoMAS 证据**:
- AUDIT S-001: README 写 `(Core × 0.5 + Gen × 0.5) × ActionabilityMultiplier`，代码实际是 `core_avg * 0.45 + gen_avg * 0.45 + (avg_action * 10) * 0.1` — 两个公式不等价
- 结果: 报告的 composite 分数含义与文档描述不符

**ECC 风险**: CLAUDE.md 和 rules/zh/ 中定义的行为可能与实际 Agent 行为不一致。例如:
- testing.md 要求 80% 覆盖率 → 但从未强制检查
- security.md 要求提交前检查 → 但无 Stop hook 执行
- code-review.md 要求 code-reviewer → 但某些场景被跳过

### 方案

**一次性审计**: 对比 rules/zh/*.md 的要求 vs 实际 Agent 行为，标记不一致项。
**定期审计**: 每月一次快速检查，确认规则仍被执行。

这是**一次性任务**，不需规则修改。执行后记录发现到 memory。

### 影响

- 工作量: 20 min (一次性)
- 输出: 一份差异报告

## AutoMAS 深度审计框架（可复用）

AutoMAS 的 AUDIT.md 定义了 7 维审计方法，可复用为 ECC 定期健康检查:

| 维度 | AutoMAS 审计内容 | ECC 对应 |
|------|------------------|----------|
| 1. 执行完整性 | 状态更新路径、代码生成机制 | Agent 调用链、状态一致性 |
| 2. 错误处理 | 信号处理、重试逻辑、静默失败 | 子 Agent 错误传播 |
| 3. 资源管理 | 磁盘、内存、并发限制 | context window、MCP server 状态 |
| 4. 实验追踪 | 结果存储、可复现性 | memory 一致性、配置漂移 |
| 5. 评分机制 | 公式一致性、边界处理 | 规则-行为一致性 |
| 6. API 限制 | 速率限制、token 预算 | MCP 工具调用限制 |
| 7. 协作协议 | 提交规范、分支策略 | Agent 间通信、状态同步 |

建议每季度用此框架做一次 ECC 健康检查。
